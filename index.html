OVERVIEW
Design Methodology

Top-Down Design
Five Phases

PHASE 01
Requirement Analysis
  • Real-time speed display in km/h
  • Wheel rotation pulses as input
  • Speed range: 0–200 km/h
  • Accuracy: ±1 km/h
  • Display refresh ≥ 5 Hz
  • Power consumption < 1 W
  • Compact, vibration-resistant
PHASE 02
System Specification
  • Pulse-based wheel sensor
  • Min. 1 pulse per rotation
  • Speed = pulses × wheel circumference
  • 12V input → 5V regulated DC
  • Numeric LCD display, km/h
  • Min. visibility: 1 meter
PHASE 03
System Architecture
  • Modular sensing → processing → output
  • Hall Effect sensor input
  • Schmitt Trigger signal conditioning
  • MCU: Timer + Interrupt + CPU
  • GPIO/I2C display interface
  • Deterministic software task flow
PHASE 04
Component Selection
  • Sensor: ATS177 Gear Tooth
  • MCU: STM32G031 (Cortex-M0+)
  • Display: Segment LCD (VA Type)
  • Regulator: 7805 + Buck Converter
  • Protection: TVS Diode + Capacitors
  • Total BOM: ₹1500–2500
PHASE 05
Integration & Testing
  • Sensor → MCU interrupt pin
  • Display via GPIO interface
  • Unit → Integration → System tests
  • Pulse count vs. speed accuracy
  • Display refresh timing test
  • Continuous operation verification

Hardware Architecture
  • Voltage Regulator (7805) powers all modules from 12V battery
  • Hall Effect Sensor pulses feed MCU interrupt pin directly
  • Microcontroller unit: Interrupt → Timer/Counter → CPU → GPIO
  • 16×2 LCD Display via I2C/GPIO interface
Justification
  • Sensing, processing, and output clearly separated
  • Improves modularity and ease of debugging
  • Real-time performance with deterministic interrupt latency
  • Noise filtering ensures clean digital pulses to MCU
SOFTWARE
Phase 3.3 — Software Architecture

Software Task Flow

01
System Start
Power-on initialisation sequence begins
02
Initialisation Module
Setup GPIO / Timer — Enable Interrupts — Init LCD Display
03
Hardware Event: Sensor Pulse
Each gear tooth passing triggers a rising edge
04
Interrupt Service Routine (ISR)
Triggered by hardware event — updates the pulse count register atomically
05
Main Control Loop (while 1)
Continuously runs Speed Compute and Display Update tasks in sequence
06
Speed Compute Task
Reads pulse count, applies formula: v = f × circumference → km/h
07
Display Update Task
Formats computed speed value and writes to LCD over GPIO/I2C
Key Design Decision
The ISR and main loop are decoupled. The ISR only increments a counter — keeping interrupt latency minimal. The main loop performs computation and display update at a safe, predictable rate ≥ 5 Hz.
Justification
Software modularisation ensures deterministic execution and real-time performance, which is essential for automotive embedded systems. Separation of ISR from computation avoids race conditions.

Phase 5 — Integration & Testing

System Validation

Integration Checklist
  • Sensor connected to MCU interrupt pin (INT)
  • Display interfaced with MCU GPIO / I2C
  • Power supply shared across all modules (5V rail)
  • Firmware compiled and flashed into STM32
Test Hierarchy
  • Unit — Sensor pulse generation, display module output
  • Integration — Pulse count vs. speed accuracy, display refresh timing
  • System — Speed at different wheel speeds, continuous operation, power stability
